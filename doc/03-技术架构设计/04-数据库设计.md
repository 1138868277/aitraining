# 数据库设计

## 文档定位

本文件只描述当前项目自己的数据库设计结果，不重复数据库通用规范。

## 设计范围

- 业务实体与表关系
- 核心表清单（18+ 张表）
- 关键字段设计
- 索引方案
- 与外部服务的数据边界

## 数据实体

核心业务实体包含 9 类：

1. **字典实体**：类型域、前缀、项目线、编码字典、数据类型、二级类码、三级类码
2. **编码生成记录**：保存生成的测点编码及上下文
3. **场站字典**：新能源场站基础信息
4. **校验与修正数据**：导入的编码稽核数据、编码映射关系、编码修正记录
5. **全量测点数据**：存量测点导入数据
6. **用户操作记录**：最近筛选条件
7. **系统用户**：平台用户账号与认证信息
8. **字典审批**：新增字典申请与审批记录
9. **时序规则数据**：导入的场站/测点数据、生成的稽核规则

## 多租户数据隔离

系统采用 **Schema 级数据隔离**，每个租户拥有独立的 PostgreSQL Schema：

| 租户 ID | Schema | 说明 |
|---------|--------|------|
| admin | `liuhaojun` | 集团管理员，包含所有字典、编码记录、用户表 |
| yunnan | `yunnan` | 云南区域，包含本区域字典、编码记录 |
| fujian | `fujian` | 福建区域，包含本区域字典、编码记录 |

所有租户共享同类型表结构（`cec_new_energy_*` 表），数据通过 Schema 隔离。用户表（`cec_sys_user`）和审批表（`cec_new_energy_code_approval`）仅在 admin Schema 中存在。

## 核心表详情

| type_name | VARCHAR(100) | 类型域名称 |

### 2. 编码字典（cec_new_energy_code_dict）— 核心字典表

| 字段名 | 类型 | 说明 |
|--------|------|------|
| code_dict_id | BIGSERIAL | 主键 |
| type_domain_code | CHAR(1) | 类型域编码（F/G/Y），CHECK约束 |
| first_class_code | VARCHAR(2) | 一级类码 |
| first_class_name | VARCHAR(100) | 一级类码名称 |
| second_class_code | VARCHAR(3) | 二级类码 |
| second_class_name | VARCHAR(100) | 二级类码名称 |
| data_category_code | VARCHAR(2) | 数据分类码 |
| data_category_name | VARCHAR(100) | 数据分类名称 |
| data_code | VARCHAR(3) | 数据码 |
| data_name | VARCHAR(100) | 数据名称 |
| is_manual | CHAR(1) | 是否手动新增（'0'否, '1'是） |

该表是编码规则的核心，定义了 5 级分类体系：类型域 → 一级类码 → 二级类码 → 数据分类码 → 数据码。

**tsr_station**：场站数据（station_code, station_name）

## 关键设计说明

- **主键策略**：所有表使用 BIGSERIAL（自增序列），由数据库自动生成
- **逻辑删除**：所有业务表使用 `if_delete` 字段标记删除状态，查询时默认过滤 `if_delete = '0'`
- **无外键约束**：表间无外键约束，数据完整性由应用层保证，便于批量数据操作
- **JSONB 存储**：最近筛选条件使用 JSONB 类型存储，充分利用 PostgreSQL 的 JSON 查询能力
- **编码解析存储**：全量测点数据表将 31 位编码按位段拆分存储，支撑按各维度独立聚合统计
- **Schema 隔离**：多租户通过不同 Schema 实现数据隔离，同类型表结构一致
- **动态建表**：编码映射表、编码修正记录表、TSR 系列表由后端应用层在首次使用时通过 `CREATE TABLE IF NOT EXISTS` 自动创建
- **审批表分离**：审批记录表（`cec_new_energy_code_approval`）仅在 admin Schema 中存在，区域侧使用草稿表（`cec_new_energy_code_draft`）跟踪状态
- **TSR 独立性**：时序规则模块使用独立的 `tsr_` 前缀表，与其他业务表无数据关联
