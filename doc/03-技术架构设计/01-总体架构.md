# 总体架构

## 架构概览

- 前端：Vue 3（Composition API）+ Element Plus + Vite + TypeScript
- 后端：Node.js + Express 4 + TypeScript（ESM）
- 数据库：PostgreSQL（pg 驱动，参数化 SQL）
- 文件处理：xlsx（Excel 解析与生成）
- 工程体系：pnpm monorepo，前后端分离

## 核心架构层次

### 后端层次结构（Controller → Service → Domain）

后端遵循三层架构，每层职责明确：

**Controller 层**（路由处理器）
- 负责请求参数解析与基础校验
- 调用 Service 层执行业务逻辑
- 封装统一响应格式（`{ code, message, data, timestamp }`）
- 不包含业务逻辑，不直接操作数据库

**Service 层**
- 编排业务用例，协调多个 Domain 服务
- 管理事务边界（通过数据库事务）
- 方法命名体现业务动作（如 `generateCode`、`batchValidate`、`correctCodes`）

**Domain 层**
- 封装核心业务规则（编码生成算法、校验规则、编码修正逻辑、统计逻辑、审批分发逻辑）
- 定义业务类型与 DTO（数据传输对象）
- 承担数据访问职责（执行 SQL 查询）

### 前端层次结构（View → Service → Router）

- **View 层**：Vue 组件（页面/视图），负责 UI 呈现与用户交互
- **Service 层**：API 调用封装（Axios），响应数据解包与错误处理
- **Router 层**：Vue Router 管理页面路由与导航，含路由守卫（`beforeEach`）进行登录态检查
- **Store 层**：Pinia 状态管理，维护用户认证信息与租户上下文

### 共享层（`@cec/contracts` + `@cec/shared`）

- **contracts**：前后端共享的 TypeScript 类型定义、错误码枚举
- **shared**：通用工具函数

### 多租户层（Tenant Context）

- 租户配置定义于 `config/tenants.ts`，支持 admin（集团）/ yunnan（云南）/ fujian（福建）三个租户
- 每个租户拥有独立的 PostgreSQL Schema 实现数据隔离
- 租户上下文中间件从 JWT Token 解析用户所属租户，通过 `AsyncLocalStorage` 传递
- 数据库查询函数（`queryAsTenant`）自动使用当前租户的连接池

## 关键设计原则

### 1. 严格分层，职责单一

- Controller 仅做参数校验和响应封装，不写业务逻辑
- Service 编排业务流程，管理事务
- Domain 封装业务规则与数据访问
- 禁止层间越权访问

### 2. 响应格式统一

所有 API 响应遵循统一格式：

```typescript
// 成功
{ code: 0, message: 'success', data: {...}, timestamp: 1712300000000 }

// 错误
{ code: 'SYSTEM_ERROR', message: '系统繁忙，请稍后重试', data: null, timestamp: 1712300000000 }
```

### 3. 统一错误处理

- 定义 `AppError` 类，携带错误码（字符串标识，如 `DICT_LOAD_FAILED`）、HTTP 状态码、错误详情
- 全局错误处理中间件统一捕获异常，返回标准错误响应
- 错误码按模块分类（通用、编码生成、校验、统计分析等）

### 4. 严格遵循数据库规范

- 所有业务表包含审计字段：`creator`、`modifier`、`create_tm`、`modify_tm`、`if_delete`
- 表名、字段名使用小写下划线命名
- 采用逻辑删除（`if_delete = '1'` 表示已删除）
- 高频查询字段建立索引

### 5. 参数校验与安全

- 前后端 Controller 层对请求参数进行基础校验
- SQL 注入防护：使用 `pg` 参数化查询（`$1`, `$2` 占位符），禁止字符串拼接 SQL
- 用户认证：JWT Token 鉴权，HS256 签名，24 小时有效期

### 6. 内存缓存

- 字典树等高频读取、低频修改的数据使用内存缓存（TTL 机制）
- 草稿数据、校验任务数据存储于后端内存，服务重启后丢失

### 7. 数据源热切换

- 通过 `datasource-store` 在运行时动态替换 PostgreSQL 连接池
- 支持在系统设置页面配置、测试、切换数据源，无需重启服务

### 8. 多租户数据隔离

- 集团（admin）和区域（yunnan/fujian）使用不同数据库 Schema
- 租户上下文通过 `AsyncLocalStorage` 在请求链路中传递
- 认证信息（用户表）统一存储在 admin 租户 Schema 下

## 模块划分

系统包含 10 个功能模块，注册在 Express 路由中：

| 模块 | 路由前缀 | 职责 |
|------|---------|------|
| 认证（auth） | `/api/auth` | 用户登录、用户 CRUD、Token 签发与验证 |
| 编码生成（code-generation） | `/api/codes` | 编码生成、草稿管理、历史记录、最近条件 |
| 自动编码（auto-code） | `/api/auto-code` | Excel 导入自动字典匹配与编码生成 |
| 编码校验（code-validation） | `/api/validate` | 编码校验、纠错建议、编码解析、重复编码稽核、编码修正 |
| 字典管理（dict） | `/api/dict` | 字典树、检索、解析、手动新增、新增记录查询 |
| 字典审批（approval） | `/api/approval` | 区域提交申请、集团审批、全租户数据分发 |
| 场站管理（station） | `/api/station` | 场站 CRUD、批量操作、导出 |
| 统计分析（statistics） | `/api/statistics` | 编码生成统计、全量测点统计、编码重复检测 |
| 时序规则（tsr） | `/api/tsr` | 场站/测点数据导入、四类稽核规则生成、多格式导出 |
| 数据源管理（datasource） | `/api/datasource` | 数据源配置、测试连接、保存切换 |

## 请求处理流程

```
客户端（浏览器）
    │
    ▼
Vite 代理（开发）/ Nginx（生产）  ─── 反向代理到后端
    │
    ▼
Express 中间件链
    ├── CORS
    ├── JSON 解析（limit: 200mb）
    ├── 请求追踪 ID（x-request-id）
    ├── 请求日志（方法、路径、耗时）
    ├── 租户上下文（从 JWT Token 解析租户，设置 AsyncLocalStorage）
    │
    ▼
Controller 层
    ├── 参数校验
    ├── 调用 Service
    ▼
Service 层
    ├── 业务逻辑编排
    ├── 协调 Domain 层
    ▼
Domain 层
    ├── 业务规则执行
    ├── 数据访问（参数化 SQL）
    ▼
PostgreSQL 数据库（按租户 Schema 隔离）
```

## 文档与 AI 友好

- 根目录提供 README.md 说明项目启动、环境配置
- 每个主要模块提供独立的 controller/service/domain 文件，职责清晰
- 错误码集中定义于 `@cec/contracts`，统一管理
- API 响应格式统一，便于前端消费和调试
- 多租户配置集中管理，各模块无需感知租户切换细节
