# 鉴权与安全设计

## 应用特点与安全前提

本应用为**内部编码生成与校验工具**，具有以下特点：
- 部署在内网环境，面向特定用户群体
- 功能主要为编码生成、校验、字典管理与统计分析
- **当前版本未接入用户认证系统**，不强制用户登录
- 数据主要为编码字典、生成记录和校验结果，不含个人敏感信息
- `.env` 文件中配置了 OAuth 相关参数（`AUTH_BASE_URL`、`OAUTH_CLIENT_ID` 等），预留了未来接入统一认证的能力

## 当前安全策略

### 访问控制
- **部署环境**：应用部署在内网，通过网络层控制访问范围
- **无用户认证**：当前不实现用户登录、注册、密码管理等功能
- **无会话管理**：不维护用户会话状态，无 Token/Cookie 机制
- **无权限分级**：所有用户具有相同操作权限

### 预留的 OAuth 配置

环境变量中已配置以下认证相关参数（当前未启用）：

| 变量 | 值 |
|------|-----|
| `OAUTH_CLIENT_ID` | `aitraining` |
| `OAUTH_CLIENT_SECRET` | 已配置 |
| `OAUTH_REDIRECT_URI` | `http://localhost:3000/api/auth/callback` |
| `AUTH_BASE_URL` | `http://leaf-auth-server.dev.jinxin.cloud/` |

未来可基于此配置接入统一 OAuth 认证流程。

## 安全要求

### 1. 输入校验与数据安全

- **前端校验**：使用 Zod 对表单输入进行严格校验（编码筛选条件、Excel 文件格式）
- **后端校验**：
  - Controller 层：使用 Zod schema 校验请求参数格式
  - Service 层：对关键业务参数进行逻辑校验（如编码格式、字典值有效性）
  - 文件上传：校验文件类型（仅限 `.xlsx`/`.xls`）、大小（≤10MB）、内容格式
- **SQL 注入防护**：
  - 使用 `pg` 参数化查询（`$1`, `$2` 占位符），禁止字符串拼接 SQL
  - 对于动态排序等场景，对字段名做白名单校验

### 2. 文件安全

- **文件上传**：
  - 校验文件 MIME 类型和扩展名
  - 限制上传文件大小（10MB，全量测点导入 200MB）
  - 使用 `multer` 处理文件上传，流式读取避免内存溢出
- **文件下载**：
  - 导出文件使用 `Content-Disposition` 附件形式下载
  - 文件名进行 URL 编码，避免路径遍历

### 3. 敏感数据处理

- **日志记录**：
  - 记录请求方法、路径、响应状态码和耗时
  - 不记录请求体中的具体数据内容
  - 不打印异常堆栈中的数据库连接信息
- **配置安全**：
  - 数据库连接信息通过 `.env` 环境变量注入
  - 数据源配置持久化于 `config/datasource.json`
  - 配置文件不提交到版本控制（`.gitignore` 已排除 `.env`）

### 4. 网络安全

- **CORS 配置**：仅允许配置的前端地址跨域访问（通过 `FRONTEND_URL` 环境变量配置）
- **请求体限制**：JSON 请求体上限 200MB，防止大请求攻击
- **会话隔离**：草稿数据通过 `x-session-id` 请求头进行隔离

### 5. 错误处理安全

- **统一错误处理**：全局错误处理中间件统一捕获异常，不向客户端暴露内部错误详情
- **标准化错误响应**：所有错误返回统一格式 `{ code, message, data, timestamp }`，不包含堆栈信息

### 6. 数据库安全

- **连接池管理**：使用 `pg.Pool` 管理数据库连接，支持连接复用
- **逻辑删除**：使用 `if_delete` 字段标记删除状态，保留数据追溯能力
- **无外键约束**：数据完整性由应用层保证，降低数据库耦合风险

## 未来可扩展的安全措施

- **OAuth 2.0 认证**：接入统一认证服务，实现用户登录与 Token 验证
- **操作审计日志**：记录关键操作的操作用户、时间、操作类型
- **接口限流**：限制单 IP/会话的请求频率
- **HTTPS 强制**：生产环境启用 HTTPS
- **IP 白名单**：进一步限制可访问的客户端 IP 范围
- **权限分级**：区分普通用户和管理员角色
