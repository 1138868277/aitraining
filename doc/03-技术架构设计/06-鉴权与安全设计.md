# 鉴权与安全设计

## 应用特点

本应用为**内部编码生成与校验工具**，部署在内网环境，面向特定用户群体。已内置完整的用户认证与鉴权体系，保障系统访问安全与数据隔离。

## 认证体系

### 登录认证流程

```
用户输入用户名和密码
       │
       ▼
POST /api/auth/login
       │
       ▼
auth-domain 查询 cec_sys_user 表验证凭据
       │
       ▼
验证成功 → 更新 last_login_time → 签发 JWT Token → 返回给前端
验证失败 → 返回 401 "用户名或密码错误"
       │
       ▼
前端将 Token 存入 localStorage
后续请求通过 Authorization: Bearer <token> 携带
```

### Token 设计

- **算法**：HS256（HMAC-SHA256），使用 Node.js 内置 `crypto` 模块实现
- **结构**：`header.payload.signature`（Base64URL 编码）
- **载荷**：`{ username, tenant, displayName, region, exp }`
- **有效期**：24 小时
- **密钥**：硬编码于 `auth-service.ts` 和 `middleware.ts`（基于简单密钥，非对称加密）

### 路由守卫

前端 Vue Router 配置了全局前置守卫（`router.beforeEach`）：
- 检查 `localStorage` 中是否存在用户信息（`auth_user`）
- 如果目标路由需要认证（`meta.requiresAuth`）且未登录，跳转至 `/login`
- 如果已登录且访问 `/login`，跳转至首页 `/code-generate`

### 默认用户

系统启动时自动在 `cec_sys_user` 表中创建默认用户：

| 用户名 | 显示名 | 区域 | 密码 | 角色 |
|--------|--------|------|------|------|
| admin | 集团管理员 | 集团 | admin | 管理员 |
| yunnan | 云南区域 | 云南省 | yunnan | 区域用户 |
| fujian | 福建区域 | 福建省 | fujian | 区域用户 |

## 权限分级

### 用户角色

系统分为两个角色层级：

1. **集团管理员（admin）**
   - 仅 admin 账号拥有管理员权限
   - 可管理所有用户（新增、编辑、删除）
   - 可审批区域提交的字典新增申请
   - 可访问系统配置中的用户管理、场站管理等管理功能

2. **区域用户（yunnan / fujian）**
   - 仅能操作本租户（Schema）下的数据
   - 不可访问用户管理、场站管理等管理功能
   - 不可查看其他区域的字典新增申请

### 权限校验方式

- 后端通过 JWT Token 中的 `tenant` 字段判断用户角色
- 管理接口在 Controller 层通过 `requireAdmin()` 函数校验
- 非 admin 用户访问管理接口返回 403 "仅集团管理员可执行此操作"

## 多租户数据隔离

### 租户上下文

- 租户配置定义于 `config/tenants.ts`，每个租户拥有独立数据库 Schema
- `tenantContextMiddleware` 从 JWT Token 解析用户所属租户
- 租户上下文通过 `AsyncLocalStorage` 在请求链路中传递
- 数据库查询函数（`queryAsTenant`）自动路由到对应 Schema

### 数据隔离方式

| 数据类型 | 隔离策略 |
|---------|---------|
| 字典数据 | Schema 隔离，各区域独立维护 |
| 编码生成记录 | Schema 隔离，仅本区域可见 |
| 场站数据 | Schema 隔离 |
| 全量测点数据 | Schema 隔离 |
| 用户数据 | 统一存储在 admin Schema 的 `cec_sys_user` 表 |
| 审批数据 | admin Schema 存储全量审批记录，区域 Schema 存储本地草稿 |

## 当前安全策略

### 1. 访问控制

- **部署环境**：应用部署在内网，通过网络层控制访问范围
- **用户认证**：所有业务接口（除登录、健康检查外）均需携带有效 JWT Token
- **权限分级**：区分集团管理员和区域用户两个角色层级
- **租户隔离**：各区域用户只能操作本 Schema 下的数据

### 2. 输入校验与数据安全

- **后端校验**：
  - Controller 层：对请求参数进行基础格式校验
  - Service 层：对关键业务参数进行逻辑校验（如编码格式、字典值有效性）
  - 文件上传：校验文件类型（仅限 `.xlsx`/`.xls`）、大小限制
- **SQL 注入防护**：
  - 使用 `pg` 参数化查询（`$1`, `$2` 占位符），禁止字符串拼接 SQL
  - 对于动态查询条件，使用参数化方式构建

### 3. 文件安全

- **文件上传**：
  - 校验文件扩展名（仅 `.xlsx`, `.xls`）
  - 限制上传文件大小（一般 10MB，全量测点导入和 TSR 导入 200MB）
  - 使用 `multer` 处理文件上传，流式读取避免内存溢出
- **文件下载**：
  - 导出文件使用 `Content-Disposition` 附件形式下载
  - 文件名进行 URL 编码，避免路径遍历

### 4. 敏感数据处理

- **日志记录**：
  - 记录请求方法、路径、响应状态码和耗时
  - 不记录请求体中的具体数据内容
  - 不打印异常堆栈中的数据库连接信息
- **配置安全**：
  - 数据库连接信息通过 `.env` 环境变量注入
  - 数据源配置持久化于 `config/datasource.json`
  - 配置文件不提交到版本控制（`.gitignore` 已排除 `.env`）
  - Token 密钥为硬编码字符串（开发环境），生产环境应替换为安全密钥

### 5. 网络安全

- **CORS 配置**：仅允许配置的前端地址跨域访问（通过 `FRONTEND_URL` 环境变量配置）
- **请求体限制**：JSON 请求体上限 200MB，防止大请求攻击
- **会话隔离**：草稿数据通过 `x-session-id` 请求头进行隔离

### 6. 错误处理安全

- **统一错误处理**：全局错误处理中间件统一捕获异常，不向客户端暴露内部错误详情
- **标准化错误响应**：所有错误返回统一格式 `{ code, message, data, timestamp }`，不包含堆栈信息

### 7. 数据库安全

- **连接池管理**：使用 `pg.Pool` 管理数据库连接，支持连接复用，多租户各自维护独立连接池
- **逻辑删除**：使用 `if_delete` 字段标记删除状态，保留数据追溯能力
- **无外键约束**：数据完整性由应用层保证，降低数据库耦合风险

## 未来可扩展的安全措施

- **OAuth 2.0 认证**：当前使用内置 JWT 实现，未来可接入统一认证服务（`.env` 中已预留 OAuth 配置参数）
- **操作审计日志**：记录关键操作的操作用户、时间、操作类型
- **接口限流**：限制单 IP/会话的请求频率
- **HTTPS 强制**：生产环境启用 HTTPS
- **IP 白名单**：进一步限制可访问的客户端 IP 范围
- **密码加密存储**：当前密码以明文存储，生产环境应使用 bcrypt 等算法进行哈希加密
- **Token 刷新机制**：当前 Token 过期后需重新登录，未来可引入 Refresh Token 机制
